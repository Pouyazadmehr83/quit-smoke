# معماری پروژه «ترک‌یار»

## ۱. نگاه کلی

این پروژه به‌صورت یک **مونولیت ماژولار** در Django طراحی شده — یعنی به‌جای
نوشتن همه چیز در یک اپ بزرگ، منطق به چند اپ مجزا با مرزهای مشخص تقسیم شده
که هرکدام یک مسئولیت دارند. این انتخاب برای MVP مناسب‌تر از میکروسرویس
است (سادگی توسعه و دیپلوی)، اما مرزهای داخلی به‌گونه‌ای کشیده شده‌اند که
در آینده اگر لازم شد، جداسازی یک اپ به سرویس مستقل راحت باشد.

```
┌─────────────────────────────────────────────────┐
│                   React Frontend                  │
│        (داشبورد، چک‌این، هدف، لیدربورد)            │
└───────────────────────┬─────────────────────────┘
                         │ REST API (JWT)
┌───────────────────────▼─────────────────────────┐
│                  Django + DRF                     │
│  ┌──────────┐ ┌───────────┐ ┌────────┐ ┌────────┐ │
│  │  users   │ │ tracking  │ │ goals  │ │leader- │ │
│  │          │ │           │ │        │ │board   │ │
│  └────┬─────┘ └─────┬─────┘ └───┬────┘ └───┬────┘ │
│       └─────────────┴────────────┴──────────┘     │
│                  apps/core (فاز بعد)               │
└───────────────────────┬─────────────────────────┘
                         │
                ┌────────▼────────┐
                │   PostgreSQL     │
                └─────────────────┘
```

## ۲. چرا این تقسیم‌بندی اپ‌ها؟

| اپ | مسئولیت | چرا جدا شده؟ |
|---|---|---|
| `users` | احراز هویت + پروفایل سیگار کاربر | داده‌ی هویتی و تنظیمات کاربر، مستقل از منطق روزانه |
| `tracking` | چک‌این روزانه + محاسبه‌ی استریک/پس‌انداز | **قلب پروژه** — منطق حساس و پرتکرارترین استفاده |
| `goals` | هدف‌گذاری مالی | به `tracking` وابسته است (می‌خواند) ولی منطق مجزایی دارد |
| `leaderboard` | رتبه‌بندی کاربران | فقط Read-only روی داده‌ی `tracking`، مدل دیتابیس ندارد |
| `core` | رزرو فاز بعد (نرخ ارز، کالاها) | جداسازی از امروز تا وقتی که اضافه شود، بدون آلوده کردن اپ‌های دیگر |

نکته‌ی مهم معماری: **منطق محاسباتی همیشه در فایل `services.py` هر اپ نوشته
شده، نه در `views.py`.** این یعنی:
- منطق به‌سادگی قابل تست واحد (unit test) است بدون نیاز به HTTP request.
- اگر فردا بخواهیم همین منطق را در یک Celery task (مثلاً برای ایمیل
  روزانه) یا یک دستور مدیریتی (management command) صدا بزنیم، فقط تابع
  سرویس را import می‌کنیم؛ کد تکراری نمی‌شود.

## ۳. مدل دیتابیس

### اپ `users`

```
User (کاربر سفارشی، AbstractUser با email به‌جای username)
├── id
├── email (unique, این فیلد لاگین است)
├── display_name
├── password (هش‌شده توسط جنگو)
└── date_joined

SmokingProfile (OneToOne با User)
├── user_id (FK → User, یکتا)
├── brand_name              # مثلا "وینستون اولترا"
├── pack_price_toman        # قیمت یک پاکت به تومان
├── cigarettes_per_pack     # پیش‌فرض 20
├── cigarettes_per_day      # میانگین مصرف روزانه قبل از ترک
└── quit_start_date

# property های محاسبه‌شده (نه ستون دیتابیس):
# price_per_cigarette = pack_price_toman / cigarettes_per_pack
# daily_saving_amount = price_per_cigarette * cigarettes_per_day
```

**چرا `daily_saving_amount` ستون دیتابیس نیست؟** چون از روی سایر فیلدها
قابل محاسبه است (derived value). ذخیره‌ی مقادیر derived در دیتابیس باعث
inconsistency می‌شود اگر کاربر بعداً قیمت پاکت را عوض کند. به‌جایش هر بار
در لحظه محاسبه می‌شود (هزینه‌ی محاسباتی‌اش ناچیز است).

### اپ `tracking`

```
DailyCheckIn
├── id
├── user_id (FK → User)
├── date                    # تاریخ (unique_together با user)
├── is_smoke_free           # True = پاک بوده, False = لغزش
├── note                    # یادداشت اختیاری
└── created_at

StreakSummary (OneToOne با User - کش محاسبه‌شده)
├── user_id (FK → User, یکتا)
├── current_streak           # روزهای پاکی متوالی فعلی
├── longest_streak            # بیشترین استریک تاریخی
├── total_smoke_free_days      # کل روزهای پاکی (نه فقط استریک فعلی)
├── total_money_saved           # کل پول پس‌انداز شده (تومان)
├── last_relapse_date
├── last_checkin_date
└── updated_at
```

**چرا `StreakSummary` به‌صورت جدول جدا و نه محاسبه‌ی Real-time از
`DailyCheckIn`؟**

این یک تصمیم عمدی performance/scalability است:
- اگر کاربری ۲ سال پاک بماند، یعنی ~۷۳۰ رکورد `DailyCheckIn` دارد. محاسبه‌ی
  استریک از روی این رکوردها در هر بار باز کردن داشبورد (`GROUP BY` و
  پیمایش تاریخ‌ها) گرچه با این تعداد رکورد مشکلی ایجاد نمی‌کند، اما با
  رشد کاربران و درخواست‌های همزمان لیدربورد (`ORDER BY` روی هزاران
  کاربر) بسیار پرهزینه می‌شود.
- به‌جایش، در لحظه‌ی هر چک‌این (`record_checkin` در `services.py`) این
  خلاصه آپدیت می‌شود — یک نوشتن ساده‌ی O(1) به‌جای یک خوانش/محاسبه‌ی O(n)
  در هر بازدید از داشبورد یا لیدربورد.
- این الگو معمولاً **«Materialized Summary»** یا **«Denormalization
  برای Read-heavy workload»** نامیده می‌شود؛ کاملاً مرسوم در اپ‌های habit
  tracking و گیمیفیکیشن.

**چرا لغزش (`is_smoke_free=False`) کل `total_money_saved` را صفر نمی‌کند؟**
چون پولی که در روزهای پاک قبلی صرفه‌جویی شده، واقعاً «خرج نشده» — لغزش
فقط استریک فعلی (روزهای متوالی) را صفر می‌کند، نه دستاورد مالی تاریخی
کاربر. این یک تصمیم محصول/روان‌شناختی است: نمی‌خواهیم با صفر کردن همه‌ی
پس‌انداز، کاربر را در یک لغزش کوچک کاملاً ناامید کنیم.

### اپ `goals`

```
FinancialGoal
├── id
├── user_id (FK → User)
├── title                     # مثلا "خرید موتور هوندا"
├── target_amount_toman
├── is_active                  # هدف اصلی نمایش داده‌شده در داشبورد
├── is_achieved
├── achieved_at
└── created_at / updated_at
```

محاسبه‌ی پیشرفت (`apps/goals/services.py`) به‌صورت Real-time از روی
`StreakSummary.total_money_saved` و `SmokingProfile.daily_saving_amount`
انجام می‌شود؛ چیزی کش نمی‌شود چون هزینه‌ی این محاسبه ناچیز است (فقط یک
تقسیم).

### اپ `leaderboard`

این اپ **هیچ مدل دیتابیس ندارد** — صرفاً Query هایی روی `StreakSummary`
می‌زند (`ORDER BY current_streak DESC` و `ORDER BY total_money_saved
DESC`، هر کدام جدا، طبق تصمیم پروژه برای دو لیست مجزا). نام واقعی کاربر
(ایمیل) هیچ‌وقت نمایش داده نمی‌شود؛ فقط `display_name`.

## ۴. منطق محاسبه‌ی استریک (مهم‌ترین بخش سیستم)

فایل: `backend/apps/tracking/services.py` → تابع `record_checkin`

```
وقتی کاربر چک‌این می‌کند:
  اگر is_smoke_free == True:
    current_streak += 1
    total_smoke_free_days += 1
    total_money_saved += daily_saving_amount (از پروفایل سیگار کاربر)
    اگر current_streak > longest_streak:
        longest_streak = current_streak
  اگر is_smoke_free == False (لغزش):
    current_streak = 0
    last_relapse_date = امروز
    (توجه: total_money_saved و total_smoke_free_days دست نمی‌خورند)
```

این منطق به‌صورت کامل در `apps/tracking/tests.py` با ۶ تست واحد پوشش
داده شده (اجرا: `python manage.py test apps.tracking`)، شامل سناریوهای:
محاسبه‌ی پایه، ۱۰ روز پاکی متوالی، لغزش و حفظ پول قبلی، حفظ
longest_streak بعد از لغزش، و خطای ثبت تکراری برای یک روز.

## ۵. احراز هویت

از **JWT (JSON Web Token)** با کتابخانه‌ی `djangorestframework-simplejwt`
استفاده شده:
- `POST /api/auth/login/` → یک `access` token (اعتبار ۱ روز) و یک
  `refresh` token (اعتبار ۱۴ روز) برمی‌گرداند.
- فرانت‌اند `access` token را در هر درخواست در هدر
  `Authorization: Bearer <token>` می‌فرستد.
- اگر `access` token منقضی شود، فرانت‌اند به‌صورت خودکار با
  `refresh` token یک توکن جدید می‌گیرد (در `frontend/src/api/client.js`
  با axios interceptor پیاده‌سازی شده).

به‌جای `username`، فیلد لاگین `email` است — این یک override ساده روی
`AbstractUser` جنگو در `apps/users/models.py` است.

## ۶. فاز بعد: سیستم «با این پول چی می‌تونی بخری؟»

طبق تصمیم گرفته‌شده در طول طراحی، این بخش *عمداً* در MVP فعلی پیاده‌سازی
نشده، اما این‌طور برنامه‌ریزی شده تا اضافه کردنش راحت باشد:

### مشکل اصلی
هیچ API رسمی و پایدار برای قیمت لحظه‌ای کالاهای متنوع (تی‌شرت، موتور و
غیره) در بازار ایران وجود ندارد. آنچه واقعاً به‌صورت API در دسترس است،
نرخ ارز (دلار/یورو) است.

### راه‌حل پیشنهادی (هیبرید)
1. مدل `Product` در `apps/core/models.py`: کالاهای نمونه با
   `price_usd` (قیمت تقریبی به دلار، نه تومان).
2. مدل `ExchangeRate` در `apps/core/models.py`: کش آخرین نرخ دلار،
   آپدیت‌شده توسط یک Celery task دوره‌ای از یک API بیرونی (مثلاً
   navasan.net)، با ستون `fetched_at` و fallback به آخرین نرخ معتبر
   اگر API بیرونی در دسترس نبود.
3. قیمت نهایی نمایش داده‌شده به کاربر:
   `قیمت تومانی کالا = نرخ دلار فعلی × Product.price_usd`

   این یعنی وقتی تورم نرخ دلار را بالا می‌برد، قیمت کالاهای نمونه
   خودکار و بدون آپدیت دستی هزاران ردیف، با بازار همگام می‌ماند.
4. Endpoint جدید: `GET /api/core/affordable-items/?amount=750000` که
   لیست کالاهایی را برمی‌گرداند که با مقدار `total_money_saved` فعلی
   کاربر قابل خریدند (نزدیک‌ترین کالاهای زیر و بالای آن مقدار).

### چرا Celery + Redis برای این بخش لازم است
چون نرخ ارز باید **دوره‌ای** (مثلاً هر ۳-۶ ساعت) از API بیرونی رفرش
شود، بدون اینکه هر درخواست کاربر منتظر یک HTTP call بیرونی بماند. این
یک کار پس‌زمینه‌ی classik برای Celery beat schedule است.

## ۷. فاز بعد: ایمیل یادآوری روزانه

طبق تصمیم پروژه، فعلاً غیرفعال (`EMAIL_NOTIFICATIONS_ENABLED=False` در
`.env`). وقتی فعال شود:
- یک Celery beat task هر روز (مثلاً ساعت ۹ صبح به وقت تهران) اجرا می‌شود.
- برای هر کاربری که هنوز امروز چک‌این نکرده (`has_checked_in_today`
  در `apps/tracking/services.py` از قبل آماده است)، یک ایمیل یادآوری
  با خلاصه‌ی استریک و پول پس‌انداز شده ارسال می‌شود.
- بک‌اند ایمیل واقعی (SMTP) باید در `EMAIL_BACKEND` در `settings.py`
  جایگزین `console.EmailBackend` فعلی شود.

## ۸. تصمیمات طراحی رابط کاربری (فرانت‌اند)

- **پالت رنگی**: فیروزه‌ای عمیق (`#0E7C7B`) + آبی نفتی (`#0B4F4D`) + سفید
  (`#FAFDFD`) برای حس آرامش و اعتماد، با یک طلایی ملایم (`#D4A24C`) که
  *فقط* برای لحظه‌ی دستاورد مالی (نوار پیشرفت هدف) استفاده می‌شود — طبق
  درخواست صریح کاربر برای تمی «آرام و قابل اطمینان».
- **عنصر امضا (signature element)**: «حلقه‌ی نفَس» — یک حلقه‌ی SVG که
  دور عدد روزهای پاکی می‌چرخد و به‌آرامی pulse می‌کند؛ استعاره از نفس
  کشیدن آرام به‌جای کشیدن سیگار. این مهم‌ترین عنصر بصری صفحه‌ی داشبورد
  است.
- **این فرانت‌اند صرفاً نمونه‌ی اولیه است** برای تست کامل جریان API ها؛
  از نظر کاربر در مکالمه، بک‌اند DRF اولویت اصلی پروژه است و فرانت‌اند
  در آینده با جزئیات بیشتر (انیمیشن، صفحات بیشتر) توسعه می‌یابد.
